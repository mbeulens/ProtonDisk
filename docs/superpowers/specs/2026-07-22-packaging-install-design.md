# ProtonDisk Packaging & Install (Design)

**Date:** 2026-07-22
**Status:** Approved (brainstorming complete)
**Scope:** A user-level installer that turns the repo checkout into an installed app —
`protondisk` on PATH, a GUI application entry, and a systemd user service that auto-mounts
`~/ProtonDisk` at login.

## 1. Approach

The app depends on **system PyGObject/GTK** and **fusepy** and runs editable from this repo
(the user's permanent home for it). A system package (.deb/Flatpak) would fight those
dependencies, so packaging is an idempotent **`install.sh`** that wires everything from the
checkout, plus an `uninstall.sh`. Re-running `install.sh` updates the install.

Verification is **live** (install → service mounts → uninstall), not unit tests — this is
shell + systemd config, not testable Python logic.

## 2. Components

Installed for the current user (machine-specific paths generated by `install.sh`, not committed):

1. **`.venv-gui`** — `python3 -m venv --system-site-packages` + `pip install -e . fusepy`.
   Self-healing launchers recreate it if missing.
2. **`~/.local/bin/protondisk`** — a thin wrapper (`exec .venv-gui/bin/python -m protondisk.cli "$@"`)
   so `protondisk auth login | ls | mount | unmount | gui | version` work from any shell.
3. **GUI application entry** `~/.local/share/applications/protondisk.desktop` — from
   `packaging/protondisk.desktop.in` (Exec → `scripts/protondisk-gui`, Icon → `assets/protondisk.svg`).
4. **systemd user service** `~/.config/systemd/user/protondisk-mount.service` — from
   `packaging/protondisk-mount.service.in`; `ExecStart=<repo>/scripts/protondisk-mount %h/ProtonDisk`,
   `enable --now`. Auto-mounts at login, unmounts at logout (SIGTERM → FUSE teardown; a lazy
   `fusermount3 -uz` in `ExecStopPost` as backstop). `Restart=on-failure`, `RestartSec=15`,
   `StartLimitBurst=5`/`StartLimitIntervalSec=300` so a transient network/session hiccup retries
   without spinning forever.

## 3. New repo artifacts (committed)

- `install.sh`, `uninstall.sh` (repo root).
- `scripts/protondisk-mount` — self-healing mount launcher (mirrors `scripts/protondisk-gui`):
  ensures the venv, then `exec python -m protondisk.cli mount "${1:-$HOME/ProtonDisk}"`.
- `packaging/protondisk-mount.service.in` — service template with an `@REPO@` placeholder.
- `packaging/protondisk.desktop.in` — GUI entry template with an `@REPO@` placeholder.

`install.sh` substitutes `@REPO@` with the checkout path via `sed`.

## 4. Honest limitations

- **Auth:** auto-mount needs a valid `proton-drive` session (stored in libsecret, readable from
  the systemd user environment). If it has expired, the service fails and retries; the user runs
  `protondisk auth login` then `systemctl --user restart protondisk-mount.service`.
- **Network at login:** if the network isn't up when the service first starts, `Restart=on-failure`
  retries (user services can't reliably order after the system `network-online.target`).
- **PATH:** `~/.local/bin` must be on `PATH` for the bare `protondisk` command; `install.sh` notes
  this if it isn't.
- Notifications work from the service (the user D-Bus session is inherited).

## 5. Verification (controller, live)

Run `install.sh`; confirm `systemctl --user status protondisk-mount` is active and `~/ProtonDisk`
lists the real Drive; confirm the `protondisk` command and the GUI `.desktop` validate; then run
`uninstall.sh` and confirm the service is gone and `~/ProtonDisk` unmounted.

## 6. Versioning

Patch bumps on `dev` per commit. This is polish on top of 0.5.0; it ships in the 0.5.x line
(no minor bump required unless the user wants a tagged 0.5.x release, or it rolls into 1.0.0).
